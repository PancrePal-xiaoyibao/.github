## 📖 这是什么 · 目的与意义

### 一句话定位
**社区对外公开招募的"愿景橱窗"**——把社区未来 12-24 个月要做的 9 件大事公开摆出来，拆成任何人都能认领的具体任务，让"宏伟目标"和"你的一次 PR"直接连起来。

### 为什么需要这个通道
| 痛点 | BillBoard 的解法 |
|---|---|
| 新人想帮忙，但不知道社区在做什么级别的事 | 9 大 Pillar 一眼看到全局，对标 pancan.org SPARK |
| 路人怕"资历不够"不敢贡献 | 每个 Pillar 有 `good first issue` + 15 分钟上手路径 |
| 社区只有私域飞书群，外人进不来 | 公开 issue，无需入群即可认领 |
| 临床/研究/Web3 等不同人群找不到自己的位置 | 每个 Pillar 有「适合人群」分诊表 |
| 社区项目散落各处，没有总抓手 | 每个 Pillar 索引行展示旗舰抓手项目 |

### 与私域体系的关系（重要）
**并行独立、相互补充，不要合并：**
- 🌐 **公域（本目录）**：招募橱窗、低门槛任务池、宣传阵地 —— 任何人可见可认领
- 🔒 **私域（飞书志愿者体系）**：深度项目、敏感合作、导师制、内部知识库 —— 入群才可见

## 🔧 运维工作流（给社区开发者）

### 日常 · 修改某个 Pillar 内容

```bash
# 1. 改本地 md（用 Edit 工具或编辑器）
# 2. 验证文件非空（必须！见 AI Onboarding 血泪教训）
wc -c community-vision/03-evidence-graph.md   # 应 > 1000 bytes

# 3. 推到线上 issue（注意 issue 编号，见映射表）
gh issue edit 18 --repo PancrePal-xiaoyibao/.github \
  --body-file community-vision/03-evidence-graph.md

# 4. 立即验证 body 真的更新了（必须！）
gh issue view 18 --repo PancrePal-xiaoyibao/.github --json body -q '.body' | wc -c
# 期望输出 > 1000，若 < 100 说明推送失败或文件为空

# 5. commit + push
git add community-vision/ && git commit -m "..." && git push
```

### 季度 · 新增一个里程碑 issue（挂在某 Pillar 下）

1. 复制下方「里程碑 issue 模板」写新 md（可放 `community-vision/milestones/` 子目录）
2. `gh issue create --repo PancrePal-xiaoyibao/.github --label "pillar-N-xxx,help wanted,good first issue" --body-file xxx.md`
3. `gh project item-add 4 --owner PancrePal-xiaoyibao --url <新issueURL>` 挂到 Project
4. 在对应 Pillar issue 的里程碑表里加一行，重推 Pillar body
5. commit 模板 md

### 长期 · 新增一个 Pillar（如 Pillar 10）

1. 写 `community-vision/10-<slug>.md`（参考现有 Pillar 结构：Why / What / Milestones / 人群 / 相关资源 / 成功指标 / Mentor / 认领说明）
2. `gh label create "pillar-10-xxx" --color "..." --repo PancrePal-xiaoyibao/.github`
3. `gh issue create` 发 issue，记下编号
4. `gh project item-add 4` 挂 Project（Pillar 字段值需 web 端补选项）
5. 在 BillBoard `00-billboard.md` 加新分组表 + 顶层索引，并全文 `N→N+1` 更新 Pillar 计数
6. 重推 BillBoard body
7. 更新本 README 的映射表 + 标签表
8. （可选）更新 profile/README 的 Pillar 索引行
9. commit + push

### 🚨 血泪教训 · 绝对不要做的事

**教训 1：不要用"临时文件 + cp 覆盖"的方式批量改 markdown**
- ❌ 错误做法：写 tmpfile → `cp tmpfile original.md`（曾导致 8 个 Pillar 文件被清空，进而推空 body 到线上 issue）
- ✅ 正确做法：**直接用 Edit 工具改原文件**，或 Write 工具完整重写

**教训 2：每次 `gh issue edit --body-file` 后必须立即验证**
- ❌ 错误做法：推完就走，"以为成功了"
- ✅ 正确做法：
  ```bash
  gh issue edit 18 --repo PancrePal-xiaoyibao/.github --body-file community-vision/03-evidence-graph.md
  gh issue view 18 --repo PancrePal-xiaoyibao/.github --json body -q '.body' | wc -c
  # 输出必须 > 1000，否则回滚
  ```

**教训 3：Windows Git Bash 编码陷阱**
- ❌ Python 内联处理中文不设编码 → UnicodeDecodeError / 乱码 / 静默截断
- ❌ 用 `/tmp/xxx.json` 临时文件 → Windows 下路径不存在
- ✅ 正确做法：
  ```bash
  PYTHONUTF8=1 python << 'PYEOF'   # 强制 UTF-8
  # 临时文件用项目内路径
  with open('cache/tmp.json', encoding='utf-8') as f: ...
  PYEOF
  ```

**教训 4：改 issue body 不可逆，推空 = 灾难**
- GitHub 不保留 issue body 历史（只能从 edit events 里捞）
- 推空 body 后，**只能从 git 历史 / 本地 md 恢复**
- 所以：**本地 md 永远是最后的救命稻草，commit 要勤**

### ✅ 标准修改流程（背下来）

```bash

# Step 3: 推到线上

  --body-file community-vision/03-evidence-graph.md

```
